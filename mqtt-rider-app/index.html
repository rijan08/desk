    MQTT for a Production React Native Rider App — Staff-Engineer Implementation Guide

    Audience: SDE1 on a RN rider/delivery app (TypeScript + Redux). You have ~45 min to learn, ~4–5 hrs to build a POC, AWS IoT Core access, and the option of a Node backend.

    The single most important thing in this whole doc: MQTT gives you real-time delivery while the app is alive and connected. It does not reliably wake a backgrounded or killed app on iOS/Android. Killed-app order assignment must go through FCM/APNs. MQTT and push are complementary, not competing. Internalize this before you write a line of code — most "MQTT in mobile" disasters come from teams expecting MQTT to replace push.

    1. MQTT Crash Course (Focused)

    What MQTT actually is

    MQTT is a lightweight publish/subscribe messaging protocol over a long-lived TCP connection (or WebSocket). Instead of clients calling each other, everyone talks to a central broker. Publishers send messages to a topic (a string path); subscribers register interest in topic patterns and the broker fans messages out to them.

    The mental model that matters:

                              ┌───────────────┐
  publish "order"  ─────▶ │               │ ─────▶ subscriber A (rider app)
  (backend)               │    BROKER     │ ─────▶ subscriber B (ops dashboard)
  publish "location" ───▶ │  (router)     │ ─────▶ subscriber C (tracking svc)
  (rider app)             └───────────────┘

    Nobody knows about anybody else. The broker is the only thing everyone connects to. This decoupling is the entire point: you can add a new consumer (analytics, a live map) without touching the publisher.

    Two properties make it good for mobile:

    • One persistent connection carries unlimited topics in both directions. You don't open a connection per "feature."
    • Tiny framing. A PUBLISH packet has ~2–4 bytes of fixed overhead. Compare to an HTTP request's headers. On a flaky 3G rider phone this is the difference between working and not.

    Why delivery/rider apps use it

    A rider app needs the server to push to the device the instant something happens (new order, reassignment, cancellation), and the device to stream to the server continuously (GPS). HTTP polling is the naive solution and it's terrible here: you either poll fast (battery + server load + cost) or poll slow (latency). MQTT inverts it — the connection stays open, server pushes when it has something, device pushes GPS on the same pipe. Low battery, low bytes, low latency. That's the whole pitch.

    MQTT vs WebSockets vs FCM vs Polling

    This is the decision table you'll get asked about in design review:

    Server→device push Device→server stream Works when app killed Battery Bytes/overhead Effort
    Polling (HTTP) ❌ (latency = interval) ✅ ❌ Bad High Trivial
    WebSocket ✅ ✅ ❌ Medium Medium You build pub/sub, QoS, reconnect yourself
    FCM/APNs (push) ✅ ❌ ✅ (the only one) Great Low Medium
    MQTT ✅ ✅ ❌ Good Lowest Library handles QoS/reconnect/sessions

    The honest summary:

    • WebSocket is a raw bidirectional pipe. MQTT is a protocol on top of that idea that already solved topics, QoS, sessions, retained messages, last-will, and reconnect semantics. Choosing raw WebSocket means re-implementing MQTT badly. (Note: MQTT can itself run over WebSocket — see §4.)
    • FCM/APNs is the only transport that reaches a killed app, but it's best-effort, rate-limited, not ordered, and high-latency for high-frequency data. You cannot stream GPS over it. You must have it anyway for killed-app wakeups.
    • MQTT is your foreground real-time workhorse. FCM is your safety net. Production rider apps run both.

    Brokers

    The broker is the server that routes. You don't write it; you run one or rent one. For you: AWS IoT Core (managed broker, you have access — use it). Self-host alternatives: Mosquitto (tiny, great for local dev, single-node), EMQX (clustered, millions of connections, production-grade), HiveMQ (enterprise). More in §6.

    Topics

    A topic is a UTF-8 string with / separators, e.g. driver/123/orders. There's no registration — you publish to a topic and it exists. Design topics like a URL hierarchy. Topic structure is your API surface; get it right early (see §2).

    Rules: no leading /, don't start segments with $ (reserved — AWS uses $aws/...), keep them reasonably short (they're sent on every message and AWS meters payload in 5KB chunks).

    Publishers / subscribers

    Same client can be both. Your rider app subscribes to driver/123/orders (to receive assignments) and publishes to driver/123/location (to stream GPS). Your backend does the inverse. Roles are per-topic, not per-connection.

    QoS 0 / 1 / 2 — the one you'll think about most

    Quality of Service = delivery guarantee, chosen per publish and capped by the subscription's QoS.

    • QoS 0 — at most once ("fire and forget"). No ack, no retry. If the connection blips, the message is gone. Use for GPS location pings — losing one is fine, the next one is 3 seconds away.
    • QoS 1 — at least once. Broker retries until it gets a PUBACK. Can deliver duplicates. Use for order assignment, cancellation, status changes — you must not lose these. You handle dedup yourself (see §7).
    • QoS 2 — exactly once. 4-packet handshake, slowest, most state. Almost never worth it on mobile. AWS IoT Core does not support QoS 2 at all — it caps at QoS 1. So this decision is made for you: QoS 1 + idempotency.

    Decision: QoS 1 for everything order-related, QoS 0 for telemetry. Build dedup by orderId/messageId and you've effectively got exactly-once at the application layer, which is the correct production pattern anyway.

    Retained messages

    A retained message is "the last value, kept by the broker." When set, any new subscriber to that topic immediately gets the last retained message without waiting for a fresh publish. Use case: driver/123/status retains online/offline so a dashboard subscribing later sees current state instantly.

    Danger for orders: never retain an order assignment. A rider reconnecting hours later would receive a stale "new order" as if it just arrived. Retained + orders = ghost orders. Retain state, never events. (More in §7.)

    Last Will and Testament (LWT)

    When you connect, you can register a "will" message: "if I disconnect ungracefully, broker, please publish THIS for me." Classic use: will = publish offline (retained) to driver/123/status. If the rider's phone dies or loses signal, the broker detects the dead connection (via keep-alive timeout) and announces them offline automatically. This is how you get reliable presence without the device having to send a goodbye.

    Sessions (clean vs persistent)

    • Clean session (cleanStart: true / cleanSession): broker forgets everything on disconnect — subscriptions and queued messages gone. Fresh start each connect.
    • Persistent session (cleanStart: false + a stable clientId + sessionExpiryInterval): broker remembers your subscriptions and queues QoS-1 messages while you're offline, redelivering on reconnect.

    For a rider app: persistent session so an order published during a 10-second tunnel disconnect still reaches the rider on reconnect. On AWS IoT Core, queued QoS-1 messages drain at 10 messages/sec on reconnect. The catch: a stable clientId per device, and exactly one connection per clientId (a second connect with the same clientId kicks the first — see reconnect storms in §7).

    Keep-alive

    A timer (e.g. 30–60s). If the client sends nothing for that long, it sends a tiny PINGREQ; broker replies PINGRESP. This (a) keeps NAT/firewall holes open and (b) lets the broker detect dead clients (miss 1.5× keep-alive → broker declares you dead and fires your LWT). Shorter keep-alive = faster dead-client detection but more radio wakeups = more battery. 30–60s is the mobile sweet spot. AWS IoT Core supports up to 1200s; on a metered/battery-sensitive rider phone, ~50s is reasonable.

    Reconnects

    Connections will drop on mobile — tunnels, elevators, network handoffs, OS throttling. The library reconnects, but you own the policy: exponential backoff + jitter, cap the delay, re-subscribe (unless persistent session restores subscriptions), and flush any queued outbound publishes. A naive "reconnect immediately in a tight loop" across a fleet is how you DDoS your own broker (§7).

    Persistent connections

    The TCP/TLS connection stays open between messages. Benefit: zero per-message setup cost, instant push. Cost: the radio can't fully sleep, and the OS may kill the socket in background. This is the core tension on mobile and the reason §7 exists.

    Wildcard topics (subscriptions only)

    • + = single level. driver/+/location matches driver/123/location and driver/999/location, not driver/123/orders.
    • # = multi-level, must be last. driver/123/# matches everything under that driver.

    Wildcards are subscribe-only (you never publish to a wildcard). The rider app subscribes to its own narrow topics. Wildcards are for backend consumers (e.g. a tracking service subscribing to driver/+/location).

    Security note: never let a device subscribe to driver/# — that's every driver's data. Policy must scope each device to its own driver/{thingName}/# (§3).

    Security basics

    1. Transport: always TLS (port 8883 native, 443 for MQTT-over-WSS). Never plaintext 1883 in production.
    2. AuthN (who are you): X.509 certs, or AWS SigV4 (Cognito), or a custom authorizer Lambda that validates your existing app JWT. For a per-user mobile app, the JWT-via-custom-authorizer path is cleanest (§3).
    3. AuthZ (what can you touch): an IoT policy that restricts each device to its own topics, using policy variables like ${iot:Connection.Thing.ThingName} so one policy scopes every device to itself.

    MQTT message flow (the full handshake, QoS 1)

    CONNECT  ─────────────▶  broker validates auth + policy
   ◀───────────── CONNACK (sessionPresent: true/false)

SUBSCRIBE driver/123/orders (qos1) ─▶
   ◀───────────── SUBACK

           ... time passes, persistent connection idle ...
PINGREQ ─▶   ◀─ PINGRESP        (keep-alive)

(backend publishes an order)
   ◀───────────── PUBLISH driver/123/orders {orderId, ...}
PUBACK ─────────────▶            (you acked receipt at the MQTT layer)

(app-level) PUBLISH driver/123/ack {orderId, accepted:true} ─▶

    Note the two layers of "ack": the MQTT PUBACK (transport got it) and your application ack (rider tapped Accept). They're different. Transport delivery ≠ business acceptance.

    Real rider/order examples

    • New order: backend → driver/123/orders (QoS1) → app shows the accept/reject modal with a countdown timer.
    • Rider accepts: app → driver/123/ack (QoS1) → backend confirms assignment, or reassigns on timeout.
    • Live tracking: app → driver/123/location (QoS0, every 3–5s) → backend rule routes to the customer's tracking view.
    • Cancellation: backend → driver/123/orders/456/status (QoS1, not retained) → app dismisses the order.
    • Presence: LWT → driver/123/status offline (retained); on connect publish online (retained).

    High-level flow

    Broker --> Rules --> DB
    Assign -.->|killed-app fallback| FCMsvc -.->|push| FCMrx

    The dotted path is the fallback: assignment service publishes over MQTT and fires an FCM data/notification message. If the app is foreground+connected, MQTT wins (instant, in-app modal). If it's killed, FCM wakes it; on open, MQTT reconnects and the persistent session (or a fetch-on-resume) reconciles state.

    Topic structure

    Design once, regret never. Convention: domain/{id}/resource[/sub].

    driver/{driverId}/orders                 # inbound: new assignment (QoS1)
driver/{driverId}/orders/{orderId}/status# inbound: accepted/cancelled/picked/delivered (QoS1)
driver/{driverId}/ack                     # outbound: rider's accept/reject (QoS1)
driver/{driverId}/location                # outbound: GPS stream (QoS0)
driver/{driverId}/status                  # presence, RETAINED (online/offline), LWT target
driver/{driverId}/commands                # inbound: server pings, force-logout, config (QoS1)

# backend-only consumers (wildcards, never used by the device):
driver/+/location                         # tracking service
driver/+/status                           # ops dashboard presence

    Why driver/{id}/... and not orders/created: it lets the IoT policy scope a device to driver/${iot:Connection.Thing.ThingName}/# so a device can only ever touch its own subtree. A flat orders/created topic would be visible to every device — a security hole. Keep the rider's {id} == its IoT thing name == its authenticated subject.

    Event naming strategy

    Topics carry routing; the payload carries the event type. Keep a discriminated union:

    type = "ORDER_ASSIGNED" | "ORDER_CANCELLED" | "ORDER_STATUS_CHANGED" | "FORCE_LOGOUT" | "CONFIG_UPDATE"

    Don't encode every event as its own topic (topic explosion, §7). Few topics, typed payloads.

    Message payload examples

    // driver/123/orders  (QoS1)
{
  "type": "ORDER_ASSIGNED",
  "messageId": "msg_01H...",     // for dedup
  "orderId": "ord_456",
  "ts": 1716960000000,
  "expiresAt": 1716960030000,    // accept window
  "pickup":  { "lat": 13.04, "lng": 80.24, "name": "Anna Nagar Kitchen" },
  "drop":    { "lat": 13.01, "lng": 80.21, "name": "T. Nagar" },
  "payout":  85.00,
  "distanceKm": 4.2
}

// driver/123/ack  (QoS1, app -> server)
{ "type": "ORDER_ACK", "messageId": "msg_01J...", "orderId": "ord_456", "accepted": true, "ts": 1716960012000 }

// driver/123/location  (QoS0, high frequency)
{ "lat": 13.031, "lng": 80.233, "heading": 271, "speed": 8.3, "ts": 1716960005000 }

    Always include messageId (idempotency) and ts. Keep order payloads under 5KB (one AWS metering unit). For tracking, the customer-facing detail is fetched via the backend, not stuffed into the GPS ping.

    Auth flow (recommended for mobile)

    1. App logs in → your backend issues JWT (already exists in your app).
2. App opens MQTT-over-WSS to AWS IoT, passing the JWT to a Custom Authorizer.
3. Authorizer Lambda validates JWT → returns an IoT policy scoping the device
   to driver/{sub}/# where sub == the user id from the token.
4. Broker enforces that policy on every CONNECT/PUBLISH/SUBSCRIBE.

    This reuses your existing identity. No per-device certificate provisioning, no Cognito identity pool to babysit. (POC shortcut: Cognito or even a dev cert — §3.)

    Reconnect strategy

    delay = min(maxDelay, base * 2^attempt) + random(0, jitter)
base=1s, maxDelay=30s, jitter=±20%
on success: reset attempt=0, re-subscribe (if clean session), flush outbound queue
only reconnect while app is foreground (pause reconnects in background — §4/§7)

    Retry strategy (application level)

    • Outbound order ACK (QoS1): library retries at MQTT layer; also keep it in an outbox until you observe a server confirmation, in case the whole connection died.
    • Don't retry GPS (QoS0) — stale location is worthless.

    QoS decisions (recap)

    Topic QoS Retained Session
    orders / status / ack / commands 1 no persistent
    location 0 no —
    status (presence) 1 yes LWT

    Offline / background / killed behavior (the honest version)

    • Foreground, connected: MQTT delivers instantly. Ideal.
    • Brief disconnect (tunnel): persistent session queues QoS1 orders; redelivered on reconnect (≤10/s on AWS).
    • App backgrounded: OS will throttle/suspend the socket within seconds–minutes. Treat backgrounded ≈ disconnected. Rely on FCM here.
    • App killed: no MQTT, period. FCM/APNs is the only path. On tap/open, app cold-starts, reconnects MQTT, and reconciles (fetch active order via REST or read retained status).

    Architectural rule: MQTT for live foreground UX; push for delivery guarantees when not foreground; REST for reconciliation on resume. Three legs of the same stool.

    Scalability (preview, full detail §7)

    • One connection per rider, scoped policy, few topics, typed payloads.
    • Backend publishes via the IoT Data Plane HTTP publish API (stateless, scales horizontally) rather than holding a broker connection per backend instance.
    • Use the Rules Engine / Basic Ingest to route inbound telemetry to DynamoDB/Kinesis without paying broker fan-out.

    3. AWS IoT Core Deep Dive

    Is AWS IoT Core a good fit?

    For your POC and early production: yes. You already have access, it's a fully managed MQTT broker (no servers to run), TLS by default, QoS 0/1, persistent sessions, LWT, retained messages, fine-grained policies, and a Rules Engine to route data into the rest of AWS. Free tier (12 months) easily covers a POC.

    Caveats that will matter later:

    • No QoS 2. Fine — you're on QoS 1 + idempotency anyway.
    • Per-connection limits: 100 messages/sec or 512 KB/s per connection; 50 subscriptions per session; max 8 topics per SUBSCRIBE call. A rider only needs a handful of subscriptions, so this is comfortable — but a backend fan-in consumer is not supposed to subscribe to everything; use Rules instead.
    • Account defaults: ~10,000 inbound and ~20,000 outbound publishes/sec, ~500 connects/sec (raise via Service Quotas). Connect rate matters during reconnect storms (§7).
    • Cost grows with message volume — it's metered (next section). At "lakhs of riders streaming GPS," IoT Core can get expensive; that's when teams evaluate self-managed EMQX. For now it's the right call.

    Architecture using AWS IoT Core

    Device ⇄ (MQTT/WSS 443) ⇄ IoT Core broker. Custom Authorizer validates JWT. Rules Engine subscribes to telemetry topics server-side and routes to DynamoDB (order state), Kinesis (location stream), Lambda (assignment logic), SNS. Backend publishes order assignments via the IoT Data Plane publish HTTP API.

    Auth options compared

    Method How Best for Pain
    X.509 certificates Per-device cert, mutual TLS Fixed IoT hardware Provisioning + rotating certs for millions of phones is miserable; certs in app storage are exfiltratable
    Cognito (SigV4) Identity pool → temp AWS creds → signed WSS URL Fastest POC Another identity system to run alongside your existing auth
    Custom Authorizer (Lambda + your JWT) App passes existing JWT; Lambda returns a scoped policy Production mobile You write/maintain a Lambda

    POC: Cognito unauthenticated/authenticated pool — fastest to a green connection. Production: Custom Authorizer reusing your app's JWT — one identity system, per-user scoping, instant revocation by failing the token.

    Easiest setup for POC (do this today)

    For the absolute fastest "device connects" moment, you can even start with a certificate generated in-console (5 clicks) and a permissive dev policy, prove the pipe works, then swap auth for Cognito/custom authorizer. Don't ship the permissive policy.

    Device identity strategy

    • Thing name == your internal driverId (or a stable device id mapped to it). This lets policy variables (${iot:Connection.Thing.ThingName}) auto-scope each device. With a custom authorizer you set the principal/thing from the JWT sub.

    Topic permissions / IAM / IoT policies

    IoT policies (not IAM policies) govern device pub/sub. The golden scoped policy:

    {
  "Version": "2012-10-17",
  "Statement": [
    { "Effect": "Allow", "Action": "iot:Connect",
      "Resource": "arn:aws:iot:REGION:ACCOUNT:client/${iot:Connection.Thing.ThingName}" },

    { "Effect": "Allow", "Action": "iot:Subscribe",
      "Resource": [
        "arn:aws:iot:REGION:ACCOUNT:topicfilter/driver/${iot:Connection.Thing.ThingName}/orders",
        "arn:aws:iot:REGION:ACCOUNT:topicfilter/driver/${iot:Connection.Thing.ThingName}/orders/*",
        "arn:aws:iot:REGION:ACCOUNT:topicfilter/driver/${iot:Connection.Thing.ThingName}/commands",
        "arn:aws:iot:REGION:ACCOUNT:topicfilter/driver/${iot:Connection.Thing.ThingName}/status"
      ]},

    { "Effect": "Allow", "Action": "iot:Receive",
      "Resource": "arn:aws:iot:REGION:ACCOUNT:topic/driver/${iot:Connection.Thing.ThingName}/*" },

    { "Effect": "Allow", "Action": ["iot:Publish","iot:RetainPublish"],
      "Resource": [
        "arn:aws:iot:REGION:ACCOUNT:topic/driver/${iot:Connection.Thing.ThingName}/location",
        "arn:aws:iot:REGION:ACCOUNT:topic/driver/${iot:Connection.Thing.ThingName}/ack",
        "arn:aws:iot:REGION:ACCOUNT:topic/driver/${iot:Connection.Thing.ThingName}/status"
      ]}
  ]
}

    Note topicfilter/ for Subscribe vs topic/ for Publish/Receive — a common gotcha that silently breaks subscriptions. The ${iot:Connection.Thing.ThingName} variable means one policy scopes every device to itself — you don't write a policy per rider.

    Scaling considerations

    • Backend = stateless HTTP publish (no broker connection to scale).
    • Inbound telemetry → Basic Ingest ($aws/rules/<ruleName>) bypasses the pub/sub broker → cheaper, no fan-out cost.
    • Raise connect-rate and connection quotas before a big rollout.
    • Shard nothing in the app; AWS handles broker scale.

    Monitoring / logging

    • Enable AWS IoT logging to CloudWatch (set default log level to WARN/ERROR in prod, DEBUG while building).
    • Key CloudWatch metrics: Connect.Success/Failure/AuthError, PublishIn/Out.Success, Subscribe.Success, Throttle, Ping.Success. Alarm on AuthError spikes (bad token rollout) and Throttle (storm).
    • The MQTT test client in the console is your fastest "is the broker even getting this?" check.

    Exact AWS setup steps (POC, cert path — ~10 min)

    1. Find your endpoint: IoT Core console → Settings → copy Device data endpoint (xxxx-ats.iot.REGION.amazonaws.com). Or aws iot describe-endpoint --endpoint-type iot:Data-ATS.
    2. Create a Thing: Console → All devices → Things → Create things → Create single thing → name it your driverId (e.g. driver-123) → Auto-generate a new certificate.
    3. Download the device cert, private key, and Amazon Root CA 1. (Private key is shown once — save it.)
    4. Create the policy above (Console → Security → Policies → Create), swap REGION/ACCOUNT. For a quick dev-only test you may use "Action":"iot:*","Resource":"*" — never ship this.
    5. Attach policy → certificate, and certificate → thing. Activate the certificate.
    6. Test in console: MQTT test client → Subscribe to driver/123/orders → in another tab Publish to it → you should see the message. The pipe works.

    Setup steps (POC, Cognito path — for RN over WebSocket)

    1. Cognito → create an Identity Pool, enable the auth (or unauth, dev only) role.
    2. Attach an IAM policy to that role allowing iot:Connect/Subscribe/Receive/Publish on your scoped ARNs (same shape as above but as an IAM policy on the Cognito role).
    3. App uses AWS credentials from Cognito to sign a WSS URL (SigV4) and connect. The AWS IoT Device SDK / amazon-cognito-identity-js does the signing; or use the custom authorizer to skip signing entirely.

    Custom Authorizer (production) — the Lambda

    // authorizer.ts  — validates your existing app JWT, returns a scoped IoT policy
import { verifyAppJwt } from "./jwt"; // your existing verifier

export const handler = async (event: any) => {
  const token = event.protocolData?.mqtt?.username     // app sends JWT as MQTT username (or in token field)
             ?? event.token;
  const claims = await verifyAppJwt(token);             // throws if invalid -> connection denied
  const driverId = claims.sub;                          // the rider's id
  const region = process.env.REGION, account = process.env.ACCOUNT;
  const T = (kind: "topic" | "topicfilter", suffix: string) =>
    `arn:aws:iot:${region}:${account}:${kind}/driver/${driverId}/${suffix}`;

  return {
    isAuthenticated: true,
    principalId: driverId,
    disconnectAfterInSeconds: 86400,
    refreshAfterInSeconds: 3600,
    policyDocuments: [{
      Version: "2012-10-17",
      Statement: [
        { Effect: "Allow", Action: "iot:Connect", Resource: `arn:aws:iot:${region}:${account}:client/${driverId}` },
        { Effect: "Allow", Action: "iot:Subscribe", Resource: [T("topicfilter","orders"), T("topicfilter","orders/*"), T("topicfilter","commands")] },
        { Effect: "Allow", Action: "iot:Receive",   Resource: T("topic","*") },
        { Effect: "Allow", Action: "iot:Publish",   Resource: [T("topic","location"), T("topic","ack"), T("topic","status")] },
      ],
    }],
  };
};

    Register it: Security → Custom authorizers → Create, point at this Lambda, (optionally enable token signing). The app then connects with the authorizer name in the connect params and the JWT as credentials.

    Debugging tips

    • Connection silently fails → 90% an auth/policy mismatch. Turn on CloudWatch IoT logs at DEBUG; the log line tells you exactly which iot:Subscribe/iot:Connect was denied and on which resource ARN.
    • Subscribe "works" but no messages → you used topic/ instead of topicfilter/ in the Subscribe statement, or the publisher used a different topic case (topics are case-sensitive).
    • Connects then immediately drops → duplicate clientId (two clients, same id — they kick each other in a loop), or LWT/keep-alive misconfig.
    • Always reproduce in the MQTT test client first to isolate device vs broker.

    Best RN MQTT libraries in 2026 — and the decision

    The hard truth: mqtt.js (the canonical JS client) depends on Node's net/tls TCP primitives that don't exist in the RN runtime, so you can't just npm i mqtt and connect over TCP. You have two viable strategies:

    AWS IoT Device SDK JS v2 WSS/SigV4 partial needs polyfills in RN — Painful in RN; skip unless you love polyfilling

    Recommendation:

    • For your 4–5 hr POC: @taoqf/react-native-mqtt (MQTT over WebSocket). Pure JS, Hermes-safe, no native build headaches, and AWS IoT Core speaks MQTT-over-WSS on 443 — so it sails through corporate/mobile firewalls and pairs with the custom-authorizer JWT flow with no SigV4 signing.
    • For production battery/background on Android: migrate the transport to a native client (sp-react-native-mqtt or evaluate @ecodevstack/...) behind the same service interface you build below. Because you'll hide the client behind an adapter, this swap is a one-file change.

    The architecture below is transport-agnostic on purpose — a thin IMqttTransport adapter wraps whichever library, so the rest of your app never imports the MQTT lib directly.

    iOS setup

    • Pure-JS WS path: nothing to build. Just allow TLS (default). If you ever use plaintext WS in dev, you'd need an ATS exception — don't; use WSS.
    • Native lib path: cd ios && pod install. Add UIBackgroundModes only if you genuinely need brief background execution (you mostly shouldn't — see limits).

    Android setup

    • WS path: autolinked, nothing to do.
    • Native lib path: autolinking handles it (RN 0.60+). Add INTERNET permission (default). For background you'll fight Doze/App Standby — see §7; the right answer is FCM, not a foreground service, unless you already run one for GPS.
    • Hermes: all recommended libs are Hermes-compatible. WS path is just JS so it's automatically fine.

    Background limitations (read before promising anything)

    • iOS: suspends your app (and its sockets) shortly after backgrounding. No general "keep MQTT alive in background." Use push.
    • Android: Doze + App Standby + aggressive OEM killers (Xiaomi/Oppo/Vivo are brutal in India) will sever the socket. A foreground service can hold a connection but you need a persistent notification and a real justification (you may already have one for live GPS during an active trip — piggyback on that, only during an active order).
    • Design consequence: connect MQTT when app is active; on background, gracefully disconnect (or let it die) and lean on FCM; on foreground, reconnect + reconcile.

    Network / reconnect / app-state handling

    • NetInfo drives reconnects: don't blindly retry while offline; wait for isConnected && isInternetReachable, then reconnect with backoff.
    • AppState drives connect/disconnect: active → ensure connected; background/inactive → stop reconnect attempts (optionally disconnect).
    • Backoff + jitter prevents reconnect storms across the fleet (§7).

    Battery considerations

    • Keep-alive 45–60s, not 10s.
    • QoS 0 for GPS; batch nothing you don't need.
    • Disconnect when backgrounded; don't hold a socket the OS will kill anyway.
    • One connection, few topics.

    Memory-leak prevention

    • Singleton service (one client, ever).
    • Every .on() listener you add, you remove on cleanup.
    • Clear all timers (reconnect, keep-alive, accept-window) on disconnect/teardown.
    • Unsubscribe Redux/event listeners in component useEffect cleanups.
    • Never create a new client on re-render.

export interface IMqttTransport {
  connect(opts: ConnectOpts): Promise<void>;
  disconnect(): Promise<void>;
  subscribe(topic: string, qos: 0 | 1): Promise<void>;
  unsubscribe(topic: string): Promise<void>;
  publish(topic: string, payload: string, qos: 0 | 1, retain?: boolean): Promise<void>;
  on(ev: "message", cb: (topic: string, payload: string) => void): void;
  on(ev: "connect" | "close" | "reconnect", cb: () => void): void;
  on(ev: "error", cb: (e: Error) => void): void;
  removeAllListeners(): void;
}
export interface ConnectOpts {
  url: string; clientId: string; username?: string; password?: string;
  keepalive: number; cleanStart: boolean; will?: { topic: string; payload: string; qos: 0|1; retain: boolean };
}
    // transport/WebSocketTransport.ts  (POC, @taoqf/react-native-mqtt over WSS)
import mqtt, { MqttClient } from "@taoqf/react-native-mqtt";
import { IMqttTransport, ConnectOpts } from "./IMqttTransport";

export class WebSocketTransport implements IMqttTransport {
  private client?: MqttClient;
  private handlers: Record<string, Function[]> = {};

  connect(o: ConnectOpts): Promise<void> {
    return new Promise((resolve, reject) => {
      // We manage reconnect ourselves -> reconnectPeriod: 0
      this.client = mqtt.connect(o.url, {
        clientId: o.clientId,
        username: o.username,
        password: o.password,
        keepalive: o.keepalive,
        clean: o.cleanStart,
        reconnectPeriod: 0,
        connectTimeout: 10_000,
        will: o.will ? { topic: o.will.topic, payload: o.will.payload, qos: o.will.qos, retain: o.will.retain } : undefined,
        protocolVersion: 4, // MQTT 3.1.1 (AWS IoT supports 3.1.1 and 5)
      });
      this.client.on("connect", () => { this.emit("connect"); resolve(); });
      this.client.on("message", (t: string, p: Buffer) => this.emit("message", t, p.toString()));
      this.client.on("close", () => this.emit("close"));
      this.client.on("error", (e: Error) => { this.emit("error", e); reject(e); });
    });
  }
  disconnect() { return new Promise<void>((res) => { this.client?.end(true, {}, () => res()); }); }
  subscribe(topic: string, qos: 0 | 1) {
    return new Promise<void>((res, rej) => this.client!.subscribe(topic, { qos }, (e) => e ? rej(e) : res()));
  }
  unsubscribe(topic: string) {
    return new Promise<void>((res, rej) => this.client!.unsubscribe(topic, (e?: Error) => e ? rej(e) : res()));
  }
  publish(topic: string, payload: string, qos: 0 | 1, retain = false) {
    return new Promise<void>((res, rej) => this.client!.publish(topic, payload, { qos, retain }, (e) => e ? rej(e) : res()));
  }
  on(ev: string, cb: Function) { (this.handlers[ev] ??= []).push(cb); }
  private emit(ev: string, ...args: any[]) { (this.handlers[ev] || []).forEach((h) => h(...args)); }
  removeAllListeners() { this.handlers = {}; this.client?.removeAllListeners?.(); }
}

    Reconnect manager

    // ReconnectManager.ts
export class ReconnectManager {
  private attempt = 0;
  private timer?: ReturnType<typeof setTimeout>;
  constructor(private base = 1000, private max = 30_000, private jitter = 0.2) {}

  schedule(reconnect: () => void) {
    const expo = Math.min(this.max, this.base * 2 ** this.attempt);
    const delay = expo + expo * this.jitter * (Math.random() * 2 - 1); // ±20%
    this.attempt++;
    this.timer = setTimeout(reconnect, Math.max(0, delay));
  }
  reset() { this.attempt = 0; this.cancel(); }
  cancel() { if (this.timer) clearTimeout(this.timer); this.timer = undefined; }
}

    Subscription manager

import { IMqttTransport } from "./transport/IMqttTransport";
export class SubscriptionManager {
  private subs = new Map<string, 0 | 1>();
  constructor(private transport: IMqttTransport) {}
  async add(topic: string, qos: 0 | 1) { this.subs.set(topic, qos); await this.transport.subscribe(topic, qos); }
  async remove(topic: string) { this.subs.delete(topic); await this.transport.unsubscribe(topic); }
  /** Call after a reconnect when using clean session (AWS persistent session restores automatically). */
  async restoreAll() { for (const [t, q] of this.subs) await this.transport.subscribe(t, q); }
  clear() { this.subs.clear(); }
}

    Dedup (idempotency)

    // dedup.ts — drop duplicate QoS1 deliveries by messageId
export class Dedup {
  private seen = new Map<string, number>();
  constructor(private ttlMs = 5 * 60_000, private max = 500) {}
  isDuplicate(id: string): boolean {
    this.gc();
    if (this.seen.has(id)) return true;
    this.seen.set(id, Date.now());
    return false;
  }
  private gc() {
    const now = Date.now();
    for (const [k, t] of this.seen) if (now - t > this.ttlMs) this.seen.delete(k);
    if (this.seen.size > this.max) this.seen.delete(this.seen.keys().next().value as string);
  }
}

    The singleton service (orchestrator)

    // MqttService.ts
import { AppState, AppStateStatus } from "react-native";
import NetInfo from "@react-native-community/netinfo";
import { WebSocketTransport } from "./transport/WebSocketTransport";
import { IMqttTransport } from "./transport/IMqttTransport";
import { ReconnectManager } from "./ReconnectManager";
import { SubscriptionManager } from "./SubscriptionManager";
import { Dedup } from "./dedup";
import { topics } from "./topics";
import { InboundEvent, MqttConfig, MqttStatus, OutboundEvent } from "./types";

type Listener<T> = (v: T) => void;

class MqttService {
  private static _i: MqttService;
  static get instance() { return (this._i ??= new MqttService()); }

  private transport!: IMqttTransport;
  private cfg!: MqttConfig;
  private t!: ReturnType<typeof topics>;
  private subs!: SubscriptionManager;
  private reconnect = new ReconnectManager();
  private dedup = new Dedup();
  private status: MqttStatus = "idle";
  private wantConnected = false;
  private appActive = true;
  private netOk = true;

  private statusListeners = new Set<Listener<MqttStatus>>();
  private eventListeners = new Set<Listener<InboundEvent>>();
  private appStateSub?: { remove: () => void };
  private netSub?: () => void;

  /** Call once after login. */
  async start(config: MqttConfig) {
    this.cfg = config;
    this.t = topics(config.driverId);
    this.wantConnected = true;

    this.appStateSub = AppState.addEventListener("change", this.onAppState);
    this.netSub = NetInfo.addEventListener((s) => {
      this.netOk = !!(s.isConnected && s.isInternetReachable);
      if (this.netOk && this.appActive) this.ensureConnected();
      else this.reconnect.cancel();
    });
    await this.ensureConnected();
  }

  /** Call on logout / teardown. */
  async stop() {
    this.wantConnected = false;
    this.reconnect.cancel();
    this.appStateSub?.remove();
    this.netSub?.();
    await this.safeDisconnect();
    this.setStatus("idle");
  }

  onStatus(cb: Listener<MqttStatus>) { this.statusListeners.add(cb); cb(this.status); return () => this.statusListeners.delete(cb); }
  onEvent(cb: Listener<InboundEvent>) { this.eventListeners.add(cb); return () => this.eventListeners.delete(cb); }

  async ackOrder(orderId: string, accepted: boolean) {
    await this.publish(this.t.ack, { type: "ORDER_ACK", orderId, accepted } as OutboundEvent, 1);
  }
  async sendLocation(lat: number, lng: number, heading?: number, speed?: number) {
    await this.publish(this.t.location, { type: "LOCATION", lat, lng, heading, speed } as OutboundEvent, 0);
  }

  // ---- internals ----
  private onAppState = (s: AppStateStatus) => {
    this.appActive = s === "active";
    if (this.appActive && this.netOk) this.ensureConnected();
    else this.reconnect.cancel(); // don't fight the OS in background
  };

  private async ensureConnected() {
    if (!this.wantConnected || this.status === "connecting" || this.status === "connected") return;
    if (!this.appActive || !this.netOk) return;
    this.setStatus("connecting");
    try {
      const token = await this.cfg.getToken();
      this.transport = new WebSocketTransport();
      this.subs = new SubscriptionManager(this.transport);
      this.wireTransport();

      const ka = this.cfg.keepAliveSec ?? 50;
      const wssUrl =
        `${this.cfg.endpoint}/mqtt` +
        (this.cfg.authorizerName ? `?x-amz-customauthorizer-name=${this.cfg.authorizerName}` : "");

      await this.transport.connect({
        url: wssUrl,
        clientId: this.cfg.driverId,           // stable -> persistent session
        username: this.cfg.driverId,
        password: token,                        // JWT consumed by custom authorizer
        keepalive: ka,
        cleanStart: false,                      // persistent session
        will: { topic: this.t.status, payload: JSON.stringify({ status: "offline", ts: Date.now() }), qos: 1, retain: true },
      });

      this.setStatus("connected");
      this.reconnect.reset();
      await this.subs.add(this.t.orders, 1);
      await this.subs.add(this.t.commands, 1);
      await this.publish(this.t.status, { status: "online", ts: Date.now() } as any, 1, true);
    } catch (e) {
      this.setStatus("error");
      this.scheduleReconnect();
    }
  }

  private wireTransport() {
    this.transport.on("message", (topic, payload) => this.handleMessage(topic, payload));
    this.transport.on("close", () => { if (this.wantConnected) { this.setStatus("offline"); this.scheduleReconnect(); } });
    this.transport.on("error", () => { /* logged by observability layer */ });
  }

  private handleMessage(topic: string, raw: string) {
    let evt: InboundEvent;
    try { evt = JSON.parse(raw); } catch { return; }                 // never trust the wire
    if (!evt || typeof (evt as any).type !== "string") return;
    if ((evt as any).messageId && this.dedup.isDuplicate((evt as any).messageId)) return; // QoS1 dup guard
    this.eventListeners.forEach((l) => l(evt));
  }

  private async publish(topic: string, body: object, qos: 0 | 1, retain = false) {
    if (this.status !== "connected") return; // (GOOD-TO-HAVE: outbox + flush on reconnect)
    await this.transport.publish(topic, JSON.stringify(body), qos, retain);
  }

  private scheduleReconnect() {
    if (!this.wantConnected || !this.appActive || !this.netOk) return;
    this.setStatus("reconnecting");
    this.reconnect.schedule(() => this.ensureConnected());
  }

  private async safeDisconnect() {
    try { this.transport?.removeAllListeners(); await this.transport?.disconnect(); } catch {}
  }
  private setStatus(s: MqttStatus) { this.status = s; this.statusListeners.forEach((l) => l(s)); }
}

    7. Production Concerns

    Scaling to lakhs/millions of riders

    • Connections: AWS IoT scales the broker for you, but raise the connect-rate and max-connections quotas before launch. One connection per rider, period.
    • Publish path: backend publishes via IoT Data Plane HTTP (stateless) → scales with your API tier, not the broker.
    • Telemetry ingest: route GPS via Basic Ingest ($aws/rules/<rule>) to Kinesis → it skips broker fan-out and is cheaper. Don't have a backend service subscribe to driver/+/location; use Rules.
    • Cost reality: at high message volume IoT Core's $1/M (5KB increments) + connection-minute charges add up. Model it (next). If GPS at 1Hz × lakhs of riders blows the budget, that's the trigger to evaluate self-managed EMQX (no inbound metering) — but that's future work, not now.

    Cost back-of-envelope (so you're not surprised)

    • 1 rider connected 24/7 ≈ $0.04/yr connectivity.
    • Messages: $1 per million (5KB units). 100k riders, each sending a GPS ping every 5s for 8 active hrs/day ≈ 100k × 5,760 pings/day ≈ 576M msgs/day ≈ ~$576/day just for location ingest before order traffic. → This is exactly why you (a) lower GPS frequency when idle, (b) use Basic Ingest, (c) eventually consider EMQX at this scale. For a POC and early launch, it's pennies.

    Topic explosion

    Don't mint a topic per event type per order (driver/123/orders/456/events/picked × millions). Keep few topics, typed payloads. Broker metadata and policy size both suffer from topic sprawl. Your topic count per rider should be a small constant (~5).

    Security risks & auth

    • TLS always; never anonymous in prod (your Mosquitto dev config is dev-only).
    • Per-device scoped policy via ${iot:Connection.Thing.ThingName} — a device must never be able to subscribe to another rider's topics or to #.
    • Revocation: custom authorizer fails the JWT → connection denied. Set refreshAfterInSeconds so policy/token re-evaluates.
    • Validate every inbound payload (JSON.parse in try/catch, check type) — never trust the wire.

    Duplicate messages

    QoS 1 will duplicate. Dedup by messageId/orderId (the Dedup class). Make all handlers idempotent: assigning the same order twice should be a no-op, not two modals.

    Ordering issues

    MQTT preserves order per topic per QoS within a single connection — but reconnects, retries, and your async handlers can reorder at the app layer. Don't rely on arrival order for correctness. Carry a ts (and ideally a server sequence number) and apply last-writer-wins by ts for status transitions. Ignore a PICKED that arrives after a DELIVERED with a later ts.

    Retained message dangers

    Retain state (status: online), never events (order assignments). A retained order = a ghost order delivered to every future subscriber. If you ever retain something transient by mistake, clear it by publishing an empty retained payload to that topic.

    Stale orders

    Every order carries expiresAt. The app ignores/auto-dismisses expired orders even if delivered late (persistent session can deliver a tunnel-delayed order after its window). The backend is the source of truth for "is this order still assignable" — the app re-validates via REST on accept if in doubt.

    Reconnect storms

    A regional network blip drops 100k riders; they all reconnect at once → broker connect-rate throttling → cascading failures. Defenses: exponential backoff + jitter (in ReconnectManager — the jitter is the important part), don't reconnect in background, and respect AWS connect-rate quotas (raise them, but also smooth demand). Stable clientId avoids the self-inflicted storm where two clients with the same id repeatedly kick each other.

    Battery drain

    Keep-alive 45–60s; QoS 0 for GPS; lower GPS frequency when idle/parked; disconnect when backgrounded. Don't hold a doomed socket in background.

    Offline riders / app killed / background restrictions

    • Background: treat as disconnected; lean on FCM.
    • Killed: FCM/APNs only path. On open, reconnect + reconcile active order via REST.
    • Android OEM killers (Xiaomi/Oppo/Vivo — very relevant in India): these aggressively kill background apps; do not promise background MQTT. High-priority FCM data messages + a foreground service only during an active trip (you likely already run one for GPS) is the realistic pattern.
    • iOS: no general background sockets. Push + on-resume reconcile.

    Rate limiting

    Per-connection: 100 msg/s or 512KB/s, 50 subscriptions, 8 topics/SUBSCRIBE. You won't hit these per rider, but a buggy GPS loop publishing 50/s will get throttled — throttle GPS client-side.

    Monitoring / observability / logging

    • CloudWatch IoT metrics: Connect.Success/AuthError, Throttle, PublishIn/Out, Ping. Alarm on AuthError (bad token rollout) and Throttle (storm).
    • App-side: log connect/disconnect/reconnect with reason + duration to your analytics (Sentry/Datadog). Track "time-to-first-order-after-connect" and "reconnect count per session" — these are your real health metrics.
    • Correlation: put messageId in both broker logs (via Rule → CloudWatch) and app logs so you can trace one order end-to-end.
    • Keep IoT logging at WARN in prod, DEBUG only when diagnosing.

    8. MQTT in Real Delivery Apps (probable architectures)

    Public engineering material is thin and these are informed inferences, not confirmed internals — treat as "how a staff engineer would expect it to be built," not gospel.

    • Uber: historically built bespoke real-time infra; their open-source Ringpop/edge stack and a custom protocol (RAMEN) for server-pushed updates are well documented in their eng blog era. The pattern: a persistent connection for live updates (trip state, driver location) + push for wakeups. Not classic MQTT, but the same shape: persistent pub/sub-ish channel + push fallback.
    • Swiggy / Zomato / Rapido / Dunzo: delivery-heavy Indian apps almost certainly run a hybrid: a persistent real-time channel (MQTT or WebSocket) for foreground order assignment + live tracking, and FCM high-priority data messages as the killed/background wakeup (essential given Indian Android OEM background-kill behavior). Order assignment is often server-driven with an accept timer, exactly the modal flow you're building. Location ingest is high-volume and typically lands in a stream (Kafka/Kinesis) behind the channel.

    Why MQTT may or may not be used

    • For MQTT: lowest battery/bytes on flaky networks (huge in India), built-in QoS/sessions/LWT, one connection many topics. Great fit for rider GPS + assignment.
    • Against / instead: some teams prefer WebSocket + their own framing (more control, easier to multiplex with existing HTTP infra, no IoT-broker lock-in), or gRPC streaming. At extreme GPS volume, managed MQTT cost pushes teams to self-hosted EMQX or a Kafka-fronted custom channel.

    The universal pattern (whatever the wire protocol)

    Foreground real-time channel (MQTT/WS)  →  instant assignment + live tracking
        +  Push (FCM/APNs)              →  killed/background wakeup (mandatory)
        +  REST                          →  reconcile state on resume / source of truth
        +  Stream (Kafka/Kinesis)        →  location & events for downstream systems

    You are building exactly the first leg correctly, with the second and third as planned fallbacks. That's the production shape.

    9. Migration Strategy for My Existing App

    You already have notifications, modals, trip/order assignment flows, Redux, timers, real-time UI. The goal is to slot MQTT under the existing UI without rewriting it.

    Phased, minimal-risk rollout

    1. Phase 0 — Adapter behind a flag. Build the §4 service. It dispatches the same Redux actions your existing push/poll path already dispatches (orderAssigned, etc.). The UI doesn't know the source changed. Ship dark (flag off).
    2. Phase 1 — Shadow mode (internal). Enable MQTT for internal/test riders. MQTT and the existing path both run; log discrepancies (did MQTT deliver the same order, faster?). No user impact.
    3. Phase 2 — MQTT primary, push fallback (small %). For flagged users, MQTT drives foreground assignment; FCM still fires for killed-app. Dedup so a rider who gets both (MQTT + push) sees one modal. Watch metrics.
    4. Phase 3 — Ramp. 1% → 5% → 25% → 100% by region (start outside your highest-volume city). Keep push fallback forever.
    5. Phase 4 — Deprecate polling (if you had any) once MQTT+push proves out.

    Feature-flag strategy

    • Server-driven flag mqtt_enabled (per user/region/app-version), read at login.
    • if (flags.mqtt_enabled) getMqttService().start(cfg) — otherwise the old path runs untouched.
    • Kill switch: flipping the flag off must cleanly stop() MQTT and fall back, no restart.

    Fallback strategy

    • MQTT down/erroring → status offline → FCM remains the delivery guarantee; optionally a low-frequency REST poll only while MQTT is unhealthy.
    • Dedup across MQTT and FCM by orderId/messageId.

    Rollback plan

    • Flag off → service stops, old path resumes. No app update needed.
    • Because the service only emits the existing Redux actions, rollback is purely "stop the new producer."

    Testing plan

    • Unit: ReconnectManager backoff math, Dedup, payload parsing/validation, topic builders.
    • Integration: against local Mosquitto — connect, subscribe, receive, ACK, reconnect, persistent-session redelivery.
    • E2E on device: airplane-mode reconnect, background→foreground, killed→FCM→open→reconcile, duplicate delivery.
    • Load/soak: simulate N clients (MQTTX scripting or a Node loop) reconnecting with jitter; watch broker throttle metrics.

    QA checklist

    • Connects on login; disconnects on logout (no leaked connection).
    • Foreground receives order < 1s after backend publish.
    • Accept → backend sees ACK; reject/timeout → backend reassigns.
    • Airplane mode 10s → reconnects, resubscribes, status correct.
    • Order published during offline window arrives after reconnect (persistent session).
    • Duplicate QoS1 delivery → exactly one modal.
    • Expired order (past expiresAt) delivered late → ignored.
    • Background → no battery-burning reconnect loop; FCM still wakes app.
    • App killed → FCM wakes → open → reconnect → active order reconciled.
    • No memory growth over a 1-hr soak (listeners/timers cleaned).
    • Flag off → fully reverts to existing behavior.

    10. Final Deliverables

    1) Learning summary

    MQTT is pub/sub over a persistent connection through a broker. For a rider app: QoS 1 + idempotency for orders, QoS 0 for GPS, persistent session so tunnel-blips don't lose orders, LWT + retained status for presence, 45–60s keep-alive for battery. AWS IoT Core is a good managed broker (no QoS 2; per-connection limits; metered cost). MQTT handles foreground real-time; FCM/APNs is mandatory for killed-app delivery; REST reconciles on resume. Hide the MQTT lib behind a transport adapter so WebSocket(POC)→native(prod) is a one-file swap.

    6) MQTT glossary

    Broker central router. Topic /-path message channel. Publish/Subscribe send to / register interest in a topic. QoS 0/1/2 at-most/at-least/exactly once (AWS = 0,1). Retained broker keeps last value for new subscribers. LWT message broker sends on ungraceful disconnect. Clean/Persistent session broker forgets / remembers subs + queues QoS1. Keep-alive ping interval + dead-client detection. Wildcards + one level, # multi (subscribe only). PUBACK MQTT-layer ack of a QoS1 publish. Client ID unique connection identity; duplicates kick each other. Basic Ingest AWS path that skips the pub/sub broker for cheaper ingest.

    7) Best practices

    • Few topics, typed payloads. Scope every device to its own subtree.
    • QoS1 + idempotency for orders; QoS0 for telemetry; no QoS2.
    • Persistent session + stable clientId; retain state not events.
    • Backoff + jitter; no background reconnect; disconnect when backgrounded.
    • Hide the lib behind a transport adapter. Validate every inbound payload.
    • Always pair with FCM (killed) and REST (reconcile). Carry messageId+ts+expiresAt.

    8) Common mistakes

    • Expecting MQTT to deliver to a killed app. (It can't. Use FCM.)
    • Retaining order events → ghost orders.
    • topic/ instead of topicfilter/ in Subscribe policy → silent no-messages.
    • Duplicate clientId → reconnect death-loop.
    • No jitter → fleet reconnect storm.
    • Trusting QoS1 to be exactly-once (it's not) → double modals.
    • Subscribing a device to #/another rider's topics → security + cost blowup.
    • Tight reconnect loop while offline/background → battery + broker abuse.
    • 1Hz GPS at scale without cost modeling → surprise IoT bill.
