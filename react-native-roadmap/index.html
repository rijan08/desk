// LEVEL 0 · threaded under L1CS Foundations

Why for you: You answered "I don't know" to process/thread and the fetch lifecycle. Without these, Level 4 and all of performance become memorization. Learn the minimum useful slice now — you don't need an OS degree.

The slice that matters for RN

  • Process vs thread. A process = your app with its own memory. A thread = a line of execution inside it. RN runs your JS on one thread (JS thread) and the UI on another (main/UI thread). That's why a heavy loop in JS freezes the screen — the UI thread is fine, it's waiting on the frozen JS thread. This sentence is the foundation of all RN performance reasoning.
  • CPU / RAM / storage. RAM = fast, volatile, where the running app lives (and where leaks bloat). Storage = slow, persistent (AsyncStorage, SQLite, Keystore). The speed gap explains why you cache and why disk reads block.
  • Networking — what happens on fetch(). DNS (name → IP) → TCP handshake → TLS handshake (the "S" in HTTPS — encryption) → HTTP request/response → JSON parse. Every network bug lives in one of these layers.
  • HTTP/HTTPS — status codes, headers, methods, idempotency (why GET is safe to retry but POST often isn't — ties to your retry helper).
  • Android architecture (high level): Linux kernel → ART runtime → app process → Activities. Enough for Levels 4 and 8.

// LEVEL 4 · PHASE 3React Native Internals TIMELY

Why for you: Blank (1/5) and your app is moving off the old bridge. As of April 2026, RN 0.85 removed the bridge entirely (0.76 made the New Architecture default; 0.82 disabled the old one). You touch native code, so this is core to your value.

Topics

  • Old architecture — the Bridge: JS thread and native thread communicated asynchronously over a bridge, serializing everything to JSON. That serialization + async round-trip was the bottleneck. Heavy data or frequent calls = lag.
  • New architecture — four pillars:
    • JSI (JavaScript Interface): lets JS hold direct references to C++/native objects and call them synchronously — no JSON, no bridge.
    • Fabric: the new rendering system; concurrent; UI stays responsive during heavy JS.
    • TurboModules: native modules loaded lazily and called via JSI (replaces old async native modules).
    • Codegen: generates the type-safe glue between JS and native from your specs.
  • Hermes: the default JS engine; ahead-of-time bytecode → faster startup, lower memory. The New Architecture depends on it.
  • Native modules & native UI components: how JS reaches Android (Kotlin/Java). You've done this "only with AI" — now learn the actual shape.
  • Threading model: JS thread, UI/main thread, native modules thread, and (Reanimated) the UI thread running worklets.

// LEVEL 6 · PHASE 3Performance Engineering YOUR BIGGEST PRACTICAL GAP

Why for you: You've never opened a profiler and you fix by guessing. The senior difference isn't more tricks — it's measuring before and after. This is where you stop guessing.

Topics

  • Three threads & the frame budget: 60fps = ~16ms/frame. Drop a frame → jank. Diagnose which thread is the culprit: JS-thread jank (heavy logic, slow renders) vs UI-thread jank (heavy native layout/animation).
  • Profiling tools: React DevTools Profiler (re-renders), Hermes profiler (JS), Android Studio Profiler (CPU/memory/network), Perfetto/systrace (frame-level).
  • Re-render storms (from L3) — measure, don't assume.
  • Memory leaks: retained listeners, timers not cleared, closures holding references. Find them with the memory profiler.
  • Startup / cold-start: Hermes, inline requires, lazy screens, reducing JS bundle parse time.
  • Lists: why FlashList beats FlatList (cell recycling) — you reach for it but didn't know the mechanism.
  • Animations: why a JS-driven animation drops frames and a UI-thread (Reanimated worklet) one doesn't — that's the whole point of worklets: they run on the UI thread so they don't wait on a busy JS thread.

// LEVEL 5 · PHASE 4Advanced React Native

Why for you: This is where your delivery-app domain makes everything concrete. You've done almost none of this hands-on (offline = never, MQTT = incomplete, security = Redux-confused).

Topics & what to actually learn

  • Navigation: React Navigation — JS stack vs native stack. Native stack (@react-navigation/native-stack) uses the platform's native navigation primitives → better performance and native gestures/transitions; JS stack is implemented in JS → more customizable, historically slower. Learn when to use which.
  • Deep linking & universal links — opening a specific order from a notification/URL.
  • Offline-first architecture: local source of truth (SQLite / WatermelonDB / MMKV), optimistic updates, a sync queue, and conflict resolution on reconnect (last-write-wins vs server-authoritative vs CRDTs). Core senior skill — start here.
  • Background processing — foreground service for live location, WorkManager for deferred sync.
  • Push notifications — FCM, notification vs data messages, handling taps → deep link.
  • Real-time (MQTT): HTTP = request/response, client pulls. WebSockets = persistent full-duplex TCP, either side pushes. MQTT = a lightweight pub/sub protocol (often over TCP/WebSocket) for many low-bandwidth, intermittently-connected clients — ideal for fleets of delivery riders publishing location. Learn the broker/topic/QoS model.
  • Security: tokens in Keystore-backed storage (react-native-keychain), not Redux/AsyncStorage; TLS & certificate pinning; biometrics; encryption at rest; app integrity (Play Integrity); obfuscation.
  • OTA updates: Expo Updates / CodePush ship JS-bundle + asset changes only; cannot change native code, permissions, or native dependencies — those need a store release.
  • Monitoring: Sentry/Crashlytics, breadcrumbs, source maps, performance traces.

How you'll know you're on it

You're solving problems no one assigned you, your designs influence other teams, and your impact is measured in what others shipped because of you. Revisit this in ~18 months.

// VERIFYThe master verification test

Beyond per-level checkpoints, you've reached senior when you can — from memory, out loud, in under 5 minutes each:

  1. Trace fetch() through every network layer.
  2. Predict any event-loop puzzle's output.
  3. Name all 5 render triggers and explain reconciliation + the key bug.
  4. Explain JSI / Fabric / TurboModules and why the old bridge was slow.
  5. Open a profiler, find a perf problem, and prove the cause.
  6. Design a live-tracking feature stating every trade-off.
  7. Store a token correctly and explain why Redux/AsyncStorage is wrong.
  8. Write a design doc and an RCA that separates symptom from cause.
If you can do all 8 cold, you have what the title requires.

// PLANSStudy plans

Daily plan — your 1 hour, 5 days/week

(Keep weekends free to avoid burnout.)

  • 0–10 min: review yesterday — re-explain one concept out loud from memory (highest-ROI study habit).
  • 10–40 min: the day's main learning — one video segment or one doc section, not both.
  • 40–60 min: apply it — small exercise, or 20 min on the current mini-project. Never a learn-only day; always touch code.
  • End each day writing one sentence: "Today I learned X, which explains why Y in my app."

Weekly plan

  • Mon–Thu: new material + exercises.
  • Fri: integration — work on the level's mini-project only.
  • Weekend (optional, no guilt): if motivated, one longer session for a talk or reading; otherwise rest.
  • End of week: can you pass this week's slice of the checkpoint? If no, next week repeats — advance on mastery, not on schedule.

6-month plan (realistic pace)

  • Month 1–1.5: L1 (JavaScript) + L0 threaded in. → You can explain the event loop, closures, GC.
  • Month 2–2.5: L3 (React internals). → You measure re-renders, understand reconciliation.
  • Month 3–4: L2 (TypeScript) — convert real files as you go. → Typed app code, generic useFetch.
  • Month 4.5–6: L4 (RN internals) — ride your team's migration as the live project. → You can explain the New Architecture; you wrote a Kotlin module.
  • Throughout: start L9 habits now (one trade-off comment per review).
  • 6-month checkpoint: the "I don't know why" feeling is gone for JS, React, and RN core.

12-month plan

  • Months 1–6: as above.
  • Months 7–8: L8 (Android slice) — read your own crash traces, write modules natively.
  • Months 9–10: L6 (Performance) — three measured case studies on your real app.
  • Months 11–12: L5 (Advanced RN) begins — offline-first order list + MQTT live-location demo.
  • Throughout: one design doc + one real RCA at work (L9).
  • 12-month checkpoint: you profile-and-prove, you've shipped offline + real-time features, you write design docs. This is the senior bar.

24-month mastery plan

  • Months 1–12: as above (senior-confident).
  • Months 13–15: finish L5 (security done right, OTA, monitoring) — fix the Redux-token habit for good.
  • Months 16–20: L7 (System design) — three design docs in your delivery domain; the state-at-scale redesign.
  • Months 21–24: L9 deepens (mentor someone, own an RCA end-to-end) + L10 begins (StaffEng reading, start solving unassigned cross-team problems).
  • 24-month state: you pass all 8 master-verification tests cold, architect features with stated trade-offs, and influence beyond your own code. Go-to architect achieved; on the Staff path.

// FINALA note to you

Two habits decide whether this works, and both are free:

  1. Re-explain out loud from memory. If you can't teach it simply, you don't know it yet. This is your daily 10 minutes.
  2. Measure before you change anything. Your instinct is to swap-until-it-works. The senior reflex is hypothesis → measure → prove → fix → re-measure. Build that and the title follows.

You don't have a knowledge-quantity problem. You have a mental-models-and-rigor problem — the kind that compounds fast once you start. Begin with Level 1 tomorrow.
